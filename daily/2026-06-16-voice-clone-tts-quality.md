# Voice clone TTS 서비스 품질 개선

이번에는 사용자의 10초 정도 되는 예제 음성을 기반으로 목소리를 복제하고, 그 목소리로 5~7분짜리 동화책을 읽어주는 Voice clone 서비스를 만드는 작업을 했다.  
처음에는 "사용자 음성을 받고, 모델에 넣고, 결과 음성을 만들면 되겠지" 정도로 생각했는데 실제로는 생각보다 훨씬 까다로웠다.  
모델 성능, 생성 시간, 비용, 긴 음성에서의 발음 품질, 그리고 마지막에 이상한 추임새가 붙는 문제까지 여러 문제가 같이 나왔다.

> Voice clone 서비스는 단순히 TTS 모델을 붙이는 작업이라기보다, 입력 음성 품질과 텍스트 전처리, 청크 분할, 후처리까지 같이 관리해야 하는 작업에 가까웠다.

## 1. 처음에는 생성 시간부터 발목을 잡았다

처음 사용하던 모델은 음성 하나를 생성하는 데 평균 20분 정도가 걸렸다.  
서비스에서 생성해야 하는 음성은 5~7분짜리 동화책 음성이었고, 사용자마다 매번 생성해야 하는 구조라서 이 시간은 꽤 치명적이었다.

시간만 문제였으면 기다리게 하면 되지 않나 싶을 수도 있는데, 실제로는 서버 비용과 바로 연결됐다.  
AWS 서버 사용료가 한 달에 100만원 정도 나오는 구조였고, 생성 시간이 길다 보니 실제 서비스에서 이윤이 남기 어려운 상태였다.

그래서 최신 TTS 모델들을 다시 찾아봤고, 그중 Qwen 쪽에서 개발한 모델이 현재 요구사항에 꽤 잘 맞았다.  
모델을 바꾼 뒤에는 기존에 한 번 생성하는 데 20분 정도 걸리던 작업이 3분 정도로 줄었다. 체감상으로는 기능 개선이라기보다 서비스가 가능한 구조로 바뀐 것에 가까웠다.

> 모델 품질도 중요하지만, 실제 서비스에서는 생성 시간이 곧 비용이고 비용이 곧 서비스 가능성으로 이어진다.

## 2. 길게 만들수록 음성이 점점 흔들렸다

모델을 바꾸고 나서 생성 시간 문제는 많이 줄었지만, 긴 음성을 만들 때는 또 다른 문제가 나왔다.  
생성 음성이 길어질수록 문장 사이의 공백이 너무 짧아지거나, 특정 문장에서 발음이 제대로 안 되는 현상이 있었다.

동화책 음성은 그냥 문장을 읽는 것보다 호흡이 중요했다.  
문장과 문장 사이에 적당한 공백이 있어야 자연스럽고, 장면이 바뀌거나 감정이 바뀌는 부분에서는 너무 빠르게 이어지면 듣는 입장에서 피로감이 생긴다.

그래서 입력 텍스트를 한 번에 길게 넣는 방식보다는, 적절한 단위로 잘라서 모델에 넣는 방식으로 바꿨다.  
청크를 너무 크게 잡으면 모델이 뒤쪽으로 갈수록 흔들렸고, 너무 작게 잡으면 청크 사이 연결감이 어색해질 수 있어서 적당한 기준을 잡는 게 중요했다.

파이썬에서는 대략 이런 흐름으로 처리했다.

```python
chunks = split_story_text(
    story_text,
    max_length=280,
    split_by_sentence=True,
)

audio_files = []

for chunk in chunks:
    normalized_text = add_pause_hints(chunk)
    audio_path = tts_client.generate(
        reference_audio_path=reference_audio_path,
        reference_text=reference_text,
        text=normalized_text,
    )

    audio_files.append(audio_path)

result_path = merge_audio_files(
    audio_files,
    silence_between_chunks_ms=700,
)
```

여기서 핵심은 텍스트를 자르는 것만이 아니었다.  
모델이 공백을 더 자연스럽게 인식하도록 `[pause 0.6s]` 같은 pause 힌트를 텍스트에 명시해주고, 청크 사이에도 실제 무음 구간을 추가했다.

> 긴 음성에서는 모델 한 번의 성능보다, 입력을 어떻게 나누고 다시 어떻게 붙이느냐가 결과 품질에 더 크게 영향을 줬다.

## 3. 결국 텍스트도 읽기 좋게 다듬어야 했다

음성 품질을 올리려고 하다 보니 텍스트 전처리도 계속 손보게 됐다.  
일반 텍스트를 그대로 넣으면 사람이 읽을 때는 자연스러워도, 모델이 읽을 때는 발음이나 호흡이 어색해지는 경우가 있었다.

예를 들어 숫자나 기호, 줄임말 같은 것들은 모델이 예상과 다르게 읽을 수 있었다.  
그래서 최대한 실제 발음되는 형태에 가깝게 텍스트를 바꾸는 방식으로 처리했다.

```python
import re


def normalize_for_speech(text: str) -> str:
    text = text.replace("AI", "에이아이")
    text = text.replace("TTS", "티티에스")
    text = re.sub(r"\s+", " ", text)
    return text.strip()


def add_pause_hints(text: str) -> str:
    text = text.replace(". ", ". [pause 0.6s] ")
    text = text.replace("? ", "? [pause 0.7s] ")
    text = text.replace("! ", "! [pause 0.7s] ")
    return text
```

또 쉼표도 단순 문장부호가 아니라 호흡을 조절하는 도구처럼 사용했다.  
특정 문장이 너무 빠르게 읽히면 쉼표를 추가해서 숨 쉴 지점을 만들고, 감정이나 장면 전환이 필요한 부분에는 `[pause 0.6s]`, `[pause 0.7s]`처럼 명시적으로 쉬는 시간을 넣었다.

이 부분은 정답이 딱 있는 작업이라기보다는, 생성 결과를 계속 들어보면서 조정하는 작업에 가까웠다.

## 4. 제일 오래 걸린 건 이상한 추임새였다

가장 오래 걸렸던 문제는 생성 음성의 마지막이나 앞부분에 이상한 추임새가 붙는 현상이었다.  
특정 조건에서만 생기는 것처럼 보였고, 처음에는 모델 할루시네이션인지, 후처리 문제인지, 참조 음성 문제인지 판단하기가 어려웠다.

원인을 바로 알 수 없어서 순서대로 가능성을 줄여갔다.

1. 먼저 생성된 음성의 마지막만 튀는 문제인지 확인하려고 fade out을 적용해봤다. 결과가 조금 부드러워지는 경우는 있었지만, 이상한 소리가 붙는 원인 자체를 없애지는 못했다.
2. 다음으로 참조 음성 품질을 의심해서 소음 제거를 적용해봤다. 노이즈가 줄어드는 효과는 있었지만, 특정 사용자 음성에서는 여전히 추임새가 붙었다.
3. 모델이 참조 음성을 잘못 이해하는지 확인하려고 `ref_text`를 더 명확하게 적어봤다. 이 방식도 일부 케이스에서는 도움이 됐지만, 문제를 완전히 해결하지는 못했다.
4. 입력 텍스트가 너무 길어서 모델이 흔들리는지 확인하려고 청크를 더 짧게 나눠봤다. 긴 음성 품질은 좋아졌지만, 마지막 추임새 문제와는 직접적인 관련이 적었다.
5. 마지막으로 청크 사이 공백과 pause 힌트를 조정해봤다. 호흡은 자연스러워졌지만, 특정 참조 음성에서만 반복되는 이상한 소리는 계속 남았다.

이 과정을 거치면서 문제의 중심이 생성 텍스트나 청크 길이보다는 참조 음성 자체에 있다는 쪽으로 좁혀졌다.

문제 원인은 결국 사용자 참조 음성의 끝부분이었다.  
예를 들어 사용자가 "안녕하세요"라고 녹음했는데, 마지막 "요"가 자연스럽게 끝나지 않고 중간에서 잘리는 경우가 있었다. 이 상태로 모델에 넣으면 모델은 참조 음성의 마지막이 아직 덜 끝난 소리라고 판단하는 것처럼 보였다.

그 결과 새 음성을 생성할 때, 모델이 사용자가 말하다 만 마지막 음절의 끝부분까지 이어서 생성해버리는 문제가 생겼다.  
그래서 실제 생성 텍스트 앞부분이나 마지막에 이상한 추임새처럼 들리는 소리가 붙었다.

> 문제는 모델이 엉뚱한 소리를 만든다기보다, 잘린 참조 음성을 이어 말하려고 하면서 생기는 현상에 가까웠다.

## 5. 답은 참조 음성을 안정적으로 받는 쪽에 있었다

이 문제는 서버에서 후처리만으로 완전히 막기 어려웠다.  
참조 음성 자체가 중간에 잘려 있으면 모델 입력부터 불안정해지기 때문에, 앱에서 녹음 UX를 개선하는 쪽이 더 효과적이었다.

그래서 녹음 안내를 더 명확하게 바꾸고, 사용자가 중지 버튼을 눌렀을 때 바로 녹음을 끊지 않고 1초 정도 여유를 둔 뒤 종료되도록 처리했다.  
사용자 입장에서는 녹음을 멈췄다고 느끼지만, 실제 파일에는 마지막 발음이 자연스럽게 끝날 시간을 조금 더 확보해주는 방식이다.

대략 이런 느낌의 처리였다.

```python
import time


def stop_recording_with_tail(recorder) -> None:
    recorder.set_state("stopping")

    time.sleep(1)
    recorder.stop()

    recorder.set_state("stopped")
```

이 방식으로 마지막 음절이 잘리는 케이스를 꽤 줄일 수 있었다.  
완벽하게 모든 문제를 없앤 건 아니지만, 가장 자주 발생하던 원인을 앱 단계에서 줄였다는 점에서 효과가 있었다.

## 6. 후처리는 어디까지나 보조였다

fade out이나 노이즈 제거 같은 후처리도 아예 의미가 없지는 않았다.  
다만 이번 문제에서는 후처리가 주 해결책은 아니었다. 잘린 참조 음성으로 인해 모델이 이상한 소리를 이어 붙이는 경우에는, 결과물을 살짝 다듬는 것보다 입력 자체를 안정적으로 만드는 게 더 중요했다.

최종적으로는 아래 순서로 변수를 줄이는 방향으로 정리했다.

1. 앱에서 녹음 안내를 명확하게 보여준다.
2. 중지 버튼을 누른 뒤 바로 끊지 않고 약간의 여유 시간을 둔다.
3. 참조 음성의 끝부분이 너무 급하게 끊긴 경우를 최대한 줄인다.
4. 입력 텍스트는 발음되는 형태에 가깝게 정리한다.
5. 쉼표와 `[pause 0.6s]` 같은 pause 힌트로 호흡을 조절한다.
6. 청크 사이에는 실제 무음 구간을 넣는다.

이렇게 하니 문제를 완전히 없앤다기보다는, 문제가 발생할 수 있는 변수를 줄여가는 방식으로 품질을 안정화할 수 있었다.

## 마무리

이번 Voice clone 작업은 모델만 바꾸면 끝나는 작업이 아니었다.  
처음에는 모델 성능과 비용 문제가 가장 커 보였고, 실제로 Qwen 계열 모델을 적용하면서 생성 시간을 20분에서 3분 정도로 줄인 건 큰 개선이었다.

하지만 그 다음부터는 더 세밀한 품질 문제가 남았다.  
긴 음성에서 문장 사이 공백이 짧아지는 문제, 발음이 흔들리는 문제, 참조 음성 끝부분이 잘려서 이상한 추임새가 붙는 문제까지 하나씩 잡아야 했다.

정리하면 이번 작업에서 배운 건 이거였다.

> Voice clone 품질은 모델 하나로만 결정되지 않는다.  
> 참조 음성의 품질, 텍스트 전처리, 청크 분할, `[pause 0.6s]` 같은 pause 힌트, 무음 삽입, 앱 녹음 UX까지 전부 결과물에 영향을 준다.

결국 모델을 잘 고르는 것도 중요하지만, 모델이 안정적으로 일할 수 있는 입력을 만들어주는 게 그만큼 중요했다.  
이번 작업은 그런 의미에서 TTS 모델을 붙이는 작업이라기보다, 음성 생성 파이프라인 전체를 조정하는 작업에 가까웠다.
