# EC2 GPU 모델 서버 운영과 자동 업데이트 장애 대응

AI 모델을 EC2에 띄워서 서비스하는 작업을 하고 있었다.  
현재 구조에서는 GPU가 필요한 모델 서버를 EC2 G6 인스턴스 위에서 실행하고 있었는데, 문제는 비용이었다.

G6 같은 GPU 인스턴스는 모델 추론에는 좋지만, 계속 켜두면 월 비용이 꽤 크게 나온다.  
우리 서비스도 한 달에 100만원 정도의 빌링이 나올 수 있는 구조였고, 아직 트래픽이 많지 않은 상황에서 GPU 서버를 여러 대 항상 켜두는 건 확실히 부담이었다.

그래서 구조를 조금 다르게 가져갔다.  
모델 서버를 항상 켜두는 대신, 요청이 들어오면 모니터링 서버가 AI 모델 서버를 가동시키고 작업이 끝나면 다시 정리하는 방식이었다. 완전한 서버리스는 아니지만, 운영 방식만 보면 서버리스와 비슷하게 필요한 순간에만 GPU 서버를 쓰는 구조에 가까웠다.

> 이번 작업의 핵심은 GPU 서버 비용을 줄이기 위해 만든 온디맨드 구조에서, 부팅 직후 자동 업데이트 때문에 모델 서버가 재시작되던 문제를 찾고 막은 것이다.

## 1. 왜 GPU 서버를 계속 켜두기 부담스러웠나

AI 모델 서버는 일반 API 서버보다 비용 구조가 훨씬 무겁다.  
특히 GPU 인스턴스는 요청이 없어도 켜져 있는 시간만큼 비용이 계속 나간다. 트래픽이 충분히 많고 계속 추론 요청이 들어오는 서비스라면 괜찮지만, 초기 서비스처럼 요청이 듬성듬성 들어오는 상황에서는 유휴 시간이 그대로 비용이 된다.

처음에는 단순하게 생각하면 모델 서버를 계속 띄워두는 게 제일 안정적이다.  
이미 서버가 켜져 있으니 cold start도 없고, 모델도 메모리에 올라가 있어서 요청을 바로 처리할 수 있다. 하지만 실제 트래픽이 많지 않은 상태에서는 안정성보다 비용 낭비가 더 크게 보였다.

그래서 선택한 방식은 아래 흐름에 가까웠다.

```text
사용자 요청
  -> API 서버 또는 모니터링 서버
  -> AI 모델 서버 상태 확인
  -> 서버가 꺼져 있으면 EC2 인스턴스 시작
  -> 모델 서버 준비 상태 확인
  -> 추론 요청 전달
  -> 작업 완료 후 필요하면 서버 종료
```

이 방식은 EC2를 쓰지만 운영 철학은 서버리스와 비슷했다.  
항상 켜두는 대신 요청이 있을 때만 서버를 켜서 비용을 줄이는 구조다.

## 2. 서버리스와 EC2 사이에서 고민한 부분

서버리스 방식의 가장 큰 장점은 서버를 계속 켜두지 않아도 된다는 점이다.  
요청이 없을 때는 비용을 거의 쓰지 않고, 요청이 들어왔을 때만 컨테이너나 함수가 실행된다. 그래서 트래픽이 적거나 들쭉날쭉한 서비스에서는 매력적이다.

하지만 AI 모델 서버에서는 단순히 서버리스가 좋다고 말하기 어려웠다.

1. GPU 모델 추론에는 GPU 리소스가 필요하다.
2. 모델 로딩 시간이 길 수 있다.
3. 컨테이너 이미지와 모델 파일 크기가 커질 수 있다.
4. 요청 하나가 짧은 함수 실행보다 오래 걸릴 수 있다.
5. 사내 보안 규정이나 인프라 검수 범위도 같이 봐야 한다.

AWS Lambda 같은 서버리스 컴퓨팅은 컨테이너 이미지 배포는 가능하지만, 실행 시간이나 메모리 같은 제약이 있고 우리가 필요로 하는 GPU 모델 서버 형태와는 잘 맞지 않았다.  
물론 AWS 외부의 RunPod 같은 GPU 서버를 쓰는 방법도 있었지만, 그렇게 되면 사내 보안 규정, 네트워크 구성, 인프라 검수 같은 일이 추가로 생긴다.

반대로 EC2는 직접 GPU 인스턴스를 선택할 수 있고, 모델 서버를 원하는 방식으로 구성하기 쉽다.  
대신 켜져 있는 시간만큼 비용이 계속 발생한다. 그래서 우리는 EC2를 쓰되, 항상 켜두지 않고 요청이 있을 때만 기동하는 방식으로 타협했다.

> 서버리스는 유휴 비용을 줄이기 좋지만 GPU 모델 서버에는 제약이 컸고, EC2는 GPU 사용은 자유롭지만 켜둔 시간만큼 비용이 나간다.  
> 그래서 이번 구조는 EC2를 직접 서버리스처럼 운영하는 방식에 가까웠다.

## 3. 그런데 특정 요청에서만 실패가 났다

문제는 가끔 요청이 실패하는 상황이었다.  
항상 실패하는 것도 아니고, 특정 시점에만 요청이 들어오면 모델 서버가 정상적으로 처리하지 못하고 죽는 현상이 있었다.

처음에는 모델 서버 자체 문제라고 생각할 수도 있었다.  
GPU 메모리가 부족한 건지, 모델 로딩이 늦은 건지, 요청 타임아웃이 너무 짧은 건지 확인해야 했다. 그런데 로그를 보니 조금 다른 흐름이 보였다.

요청이 들어온 뒤 모델 서버가 갑자기 내려가고 있었다.  
앱 로그에는 "서버 종료 중"과 비슷한 시각의 로그가 찍혀 있었고, system 로그에서도 `fairytale.service`가 중지되거나 다시 시작되는 기록이 보였다.

처음에는 왜 요청이 들어온 타이밍에 서버가 내려가는지 이해가 잘 안 됐다.  
그런데 로그를 시간순으로 맞춰보니 서버가 켜진 직후 밀려 있던 OS 업데이트가 실행되고, 그 과정에서 서비스 재시작이 발생하고 있었다.

> 모델 서버가 요청을 처리하다가 죽은 게 아니라, 부팅 직후 자동 업데이트가 실행되면서 서비스가 재시작된 것이었다.

## 4. 원인은 unattended-upgrades와 needrestart였다

Ubuntu 서버에서는 자동 업데이트가 설정되어 있으면 `apt-daily.timer`, `apt-daily-upgrade.timer` 같은 systemd timer가 주기적으로 실행된다.  
이 자체는 좋은 기능이다. 보안 업데이트를 자동으로 적용해주기 때문이다.

하지만 우리 구조에서는 문제가 됐다.  
서버가 계속 켜져 있는 일반 서버라면 자동 업데이트가 특정 시간에 돌고 끝난다. 그런데 우리는 서버를 필요할 때만 켠다. 그러면 서버가 꺼져 있는 동안 밀려 있던 업데이트가 다음 부팅 시점에 몰아서 실행될 수 있다.

즉 흐름은 이랬다.

```text
서버가 오래 꺼져 있음
  -> 그 사이 업데이트가 밀림
  -> 요청이 들어와서 EC2 부팅
  -> 부팅 직후 apt 자동 업데이트 실행
  -> 패키지 업그레이드 진행
  -> needrestart가 관련 서비스 재시작
  -> fairytale.service 중지/시작
  -> 요청 처리 중 서버가 내려간 것처럼 보임
```

특히 `needrestart`가 설치되어 있으면 패키지 업데이트 이후 재시작이 필요한 서비스를 감지하고 재시작할 수 있다.  
이게 모델 서버가 막 요청을 처리하려는 시점과 겹치면, 사용자 입장에서는 요청이 갑자기 실패한 것처럼 보인다.

## 5. 로그는 시간순으로 맞춰봐야 했다

이 문제는 한 로그만 봐서는 바로 보이지 않았다.  
앱 로그만 보면 "왜 서버가 내려갔지?" 정도로만 보이고, apt 로그만 보면 "업데이트가 있었네" 정도로만 보인다. 두 로그의 시간을 맞춰봐야 둘이 같은 사건이라는 걸 알 수 있었다.

### fairytale 서비스가 언제 중지됐는지 확인

먼저 모델 서버 서비스가 언제 멈췄는지 확인했다.

```bash
sudo grep 'fairytale' /var/log/syslog | grep -E 'Stopping|Stopped|Starting|Finished'
```

여기서 `Stopping fairytale.service`가 `01:41:02`, `01:42:29`, `01:42:58`에 찍혀 있었다.  
이 시간이 앱 로그에 찍힌 "서버 종료 중" 시각과 초 단위로 거의 일치했다.

### 같은 시각에 apt가 무엇을 했는지 확인

그 다음 apt history를 확인했다.

```bash
sudo grep -E 'Start-Date|Commandline|Upgrade:' /var/log/apt/history.log
```

여기서는 `Commandline: /usr/bin/unattended-upgrade`가 `01:42:20`부터 `01:43:18` 사이에 연달아 실행된 기록이 보였다.  
그리고 `libgnutls30`, `libcaca0` 같은 패키지가 업그레이드된 기록도 있었다.

중요한 건 업그레이드 직후 `fairytale.service`가 중지되는 흐름이 반복됐다는 점이다.  
이걸 보고 단순 앱 장애가 아니라 OS 자동 업데이트와 서비스 재시작이 엮인 문제라고 판단할 수 있었다.

### unattended-upgrades 로그 확인

자동 업데이트 자체 로그도 확인했다.

```bash
sudo less /var/log/unattended-upgrades/unattended-upgrades.log
```

특정 시간대만 보고 싶으면 아래처럼 확인할 수 있다.

```bash
sudo grep '2026-06-12 01:4' /var/log/unattended-upgrades/unattended-upgrades.log
```

여기서 `01:43:17`에 `All upgrades installed`가 찍힌 뒤로 재시작이 멈춘 것을 확인할 수 있었다.  
즉 업데이트가 끝나기 전까지 서비스 재시작이 이어졌고, 업데이트가 끝난 뒤에는 같은 문제가 멈춘 것이다.

### needrestart 설치 여부 확인

서비스 재시작이 왜 발생했는지도 확인했다.

```bash
dpkg -l needrestart
```

`needrestart`가 설치되어 있으면 패키지 업데이트 이후 재시작이 필요한 데몬이나 서비스를 확인하고 재시작을 유도할 수 있다.  
이번 문제에서는 unattended-upgrades와 needrestart의 조합이 부팅 직후 모델 서버 재시작으로 이어진 것으로 볼 수 있었다.

### 부팅 직후에만 터지는 패턴인지 확인

마지막으로 인스턴스 켜짐/꺼짐 이력을 봤다.

```bash
last -x reboot shutdown | head -20
```

기록을 보면 `01:25`에 부팅되고, `01:40`부터 `01:43` 사이에 업그레이드와 서비스 재시작이 연달아 발생했고, 이후 `01:46`에 자체 종료되는 흐름이 보였다.

이 흐름을 보면 문제가 아무 때나 발생하는 게 아니라, 서버가 켜진 직후 밀린 업데이트가 실행될 때 발생한다는 걸 알 수 있었다.

## 6. 해결은 자동 업데이트 타이머를 끄는 쪽으로 했다

결국 해결은 자동 업데이트 타이머를 끄는 방식으로 했다.

```bash
sudo systemctl disable --now apt-daily-upgrade.timer apt-daily.timer
```

이 명령어는 두 timer를 비활성화하고, 현재 실행 중이면 같이 멈춘다.

적용 후에는 아래 명령어로 확인할 수 있다.

```bash
systemctl is-enabled apt-daily-upgrade.timer apt-daily.timer
```

둘 다 `disabled`가 나오면 일단 자동으로 업데이트가 실행되는 흐름은 막힌 것이다.

다만 자동 업데이트를 끄는 건 "업데이트를 영원히 하지 않겠다"는 뜻이 아니다.  
이건 꽤 중요한 부분이다. 자동 업데이트를 끄면 운영 중 갑자기 서비스가 재시작되는 문제는 줄일 수 있지만, 보안 패치가 밀릴 수 있다.

그래서 최소한 분기에 한 번 정도는 수동 패치 창을 잡는 방식으로 운영해야 한다고 봤다.

```bash
sudo apt update
sudo apt upgrade
```

업데이트 후에는 바로 끝내지 말고 TTS 또는 모델 서버 동작 확인까지 같이 해야 한다.  
미루면 미룰수록 한 번에 올라가는 패키지 폭이 커지고, 그만큼 깨질 확률도 올라간다.

> 자동 업데이트를 끄는 건 안정성을 위해 필요한 선택이었지만, 보안 패치를 포기하는 선택은 아니다.  
> 자동으로 하던 일을 사람이 정해진 점검 시간에 책임지고 하는 방식으로 바꾼 것에 가깝다.

## 7. 다음에 같은 문제가 날지 미리 보는 방법

같은 문제가 다시 발생할 가능성이 있는지도 간단히 확인할 수 있다.

```bash
apt list --upgradable 2>/dev/null | wc -l
```

결과가 `1`보다 크면 업데이트할 패키지가 남아 있다는 뜻이다.  
자동 업데이트가 켜져 있다면 다음 부팅 시점에 다시 업데이트가 실행될 후보가 있는 셈이다.

물론 이 값이 크다고 무조건 장애가 나는 건 아니다.  
하지만 우리처럼 요청이 들어올 때만 GPU 서버를 켜는 구조에서는 부팅 직후 실행되는 백그라운드 작업이 요청 처리와 겹칠 수 있기 때문에 위험 신호로 볼 수 있다.

## 마무리

이번 문제는 겉으로 보면 모델 서버 장애처럼 보였다.  
요청이 들어왔고, 그 직후 서버가 내려갔으니 처음에는 모델 로딩이나 GPU 메모리 문제를 의심할 수밖에 없었다.

하지만 실제 원인은 모델 코드가 아니라 서버 운영 방식에 있었다.  
비용을 줄이기 위해 EC2 GPU 서버를 필요할 때만 켜는 구조를 만들었고, 그 결과 부팅 직후 실행되는 자동 업데이트가 실제 요청 처리 시간과 겹치게 됐다.

이번 작업에서 배운 건 이거였다.

> 비용 최적화를 위해 서버를 껐다 켜는 구조를 만들면, 부팅 직후 실행되는 모든 작업도 같이 운영 대상이 된다.  
> 모델 서버만 잘 떠 있는지 볼 게 아니라, OS 업데이트, systemd timer, needrestart, 서비스 재시작까지 같이 봐야 한다.

결국 서버리스처럼 운영하려면 애플리케이션만 서버리스처럼 만들면 안 된다.  
서버가 켜지는 순간 어떤 백그라운드 작업이 같이 도는지까지 통제해야 진짜 안정적으로 운영할 수 있다.
